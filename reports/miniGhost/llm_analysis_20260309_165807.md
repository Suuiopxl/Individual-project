## 1. `__mg_stencil_comps_mod_MOD_mg_stencil_2d5pt`

### 1) 功能摘要
二维 5 点计算模板（Stencil）在三维 GRID 数据上进行加权平均并写回。

### 2) 计算特征
**计算密集型**，但含有大量内存访问且读写数组。

### 3) 数据依赖分析
迭代之间无依赖，且网格点更新采用双缓冲（WORK 和 GRID），可完全并行。

### 4) GPU 加速可行性评分
9/10

### 5) 推荐 GPU 化方案
**OpenACC 或 CUDA**

### 6) 预期加速比
5～15 倍（视数据传输和硬件差异）

### 7) 实现难度
中

### 8) 具体建议
- 将 GRID 和 WORK 数据拷贝到 GPU 端（使用 OpenACC data region 或 CUDA 全局内存）
- 使用 `parallel loop collapse(3)` 并行执行三层循环
- 计算完成后更新 GRID
- 避免过多数据传输，通过持久数据驻留来降低开销
- 若用 CUDA，设计三维网格与块映射

伪代码（OpenACC）：
```fortran
!$acc data copy(GRID), create(WORK)
!$acc parallel loop collapse(3)
do k=1,nz
  do j=1,ny
    do i=1,nx
      WORK(i,j,k) = (GRID(i-1,j,k) + GRID(i,j-1,k) + GRID(i,j,k) + GRID(i,j+1,k) + GRID(i+1,j,k))/5.0
    end do
  end do
end do

!$acc parallel loop collapse(3)
do k=1,nz
  do j=1,ny
    do i=1,nx
      GRID(i,j,k) = WORK(i,j,k)
    end do
  end do
end do
!$acc end data
```

## 2. `__mg_sum_grid_mod_MOD_mg_sum_grid`

### 1) 功能摘要
计算 GRID 的所有元素的全局和。

### 2) 计算特征
涉及大量的全数组遍历，是**内存访问密集型**，且包含**MPI通信**（Allreduce）。

### 3) 数据依赖分析
数组求和无依赖，局部可并行（利用 OpenMP reduction），但 MPI Allreduce 存在通信依赖。

### 4) GPU 加速可行性评分
7/10

### 5) 推荐 GPU 化方案
**OpenACC + MPI**

### 6) 预期加速比
3～8 倍（计算加速明显，但 MPI 通信仍是瓶颈）

### 7) 实现难度
中

### 8) 具体建议
- 将 GRID 复制到 GPU，利用 GPU 并行执行求和（CUDA Reduce 或 OpenACC reduction）
- 计算结果从 GPU 拷贝回 CPU
- 继续使用 MPI_Allreduce 做全局通信（通信仍在 CPU）
- 可尝试 GPU Direct RDMA 来降低通信开销（高级）

伪代码（OpenACC）：
```fortran
real(kind=MG_REAL) :: LSUM
LSUM = 0.0
!$acc data copyin(GRID)
!$acc parallel loop collapse(3) reduction(+:LSUM)
do k=1,nz
  do j=1,ny
    do i=1,nx
      LSUM = LSUM + GRID(i,j,k)
    end do
  end do
end do
!$acc end data

call MPI_ALLREDUCE(LSUM, GSUM, MPI_SUM,...)
```

## 3. `__mg_flux_accumulate_mod_MOD_mg_flux_accumulate_2d5pt_3d7pt`

### 1) 功能摘要
在 GRID 的边界面上按照模板加权累计 FLUX_OUT。

### 2) 计算特征
**内存访问密集型，小量算术计算。**

### 3) 数据依赖分析
不同边界面的循环无依赖，只是将结果累加到 FLUX_OUT(IVAR)，有写依赖。

### 4) GPU 加速可行性评分
5/10

### 5) 推荐 GPU 化方案
**不推荐单独 GPU 加速；若整体迁移到 GPU，可用 OpenACC**

### 6) 预期加速比
<2 倍（因计算量极小，内存访问及累加操作有限）

### 7) 实现难度
低

### 8) 具体建议
- 该函数计算量小，频繁访问 FLUX_OUT 并涉及原子累加
- 可改写为 GPU kernel，使用原子操作或者先对边界面做块内加和再合并
- 若应用中其他步骤已GPU化，则一并迁移
- 否则不建议单独加速
