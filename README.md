# AI4Science — LLM-Driven GPU Optimization Pipeline

> **COMP3931 Individual Project — University of Leeds**
>
> Automatically discover performance bottlenecks in HPC scientific applications and accelerate them using LLM-driven GPU code transformation.

## Project Overview

This project explores whether Large Language Models (LLMs) can be used as intelligent code agents to identify performance-critical code in scientific applications and automatically transform them into GPU-accelerated versions. The pipeline covers four representative AI4Science mini-applications spanning different computational domains and programming languages.

The complete workflow is:

1. **Deploy & Profile** — Build each application, run gprof/perf profiling to identify hotspot functions.
2. **LLM Analysis** — Feed profiling data and source code to an LLM agent (GPT-4o / Claude) to analyze GPU acceleration feasibility.
3. **GPU Transformation** — Apply LLM-suggested optimizations: either automated directive-based patching (OpenACC) or manual CUDA kernel rewrites guided by LLM output.
4. **Benchmark & Validate** — Compare CPU vs GPU performance with correctness verification.

## Prerequisites

### Hardware

- **CPU**: Any modern x86_64 processor
- **GPU**: NVIDIA GPU with Compute Capability ≥ 5.0 (tested on RTX 4080 Laptop, SM 8.9)
- **RAM**: ≥ 16 GB recommended (LAMMPS large builds require significant memory)

### Software

- **OS**: Ubuntu 22.04+ (native Linux or WSL2 on Windows)
- **GPU Driver**: NVIDIA driver ≥ 535 (must support CUDA 12+)
- **CUDA Toolkit**: 12.0+ (tested with CUDA 13.1 via NVIDIA HPC SDK 26.1)
- **Compilers**:
  - GCC/G++/GFortran (system default, for CPU builds)
  - NVIDIA HPC SDK (`nvfortran`, `nvc`, `nvc++` for OpenACC; `nvcc` for CUDA)
- **MPI**: OpenMPI (`mpirun`, `mpifort`, `mpicxx`)
- **Build tools**: `make`, `cmake` (≥ 3.18 for LAMMPS), `jq`
- **Python**: 3.10+ with `openai` and/or `anthropic` packages
- **LLM API key**: OpenAI API key (GPT-4o) or Anthropic API key (Claude) — required only for Step 2 (LLM analysis)

## Quick Start — Full Pipeline Walkthrough

### Step 0: Clone and Set Up Environment

```bash
git clone https://github.com/Suuiopxl/Individual-project.git
cd Individual-project

# Install all dependencies, clone app source repos, create GPU copies
bash scripts/00_setup_local_env.sh

# Or set up a specific app only:
bash scripts/00_setup_local_env.sh miniGhost
```

> **Note:** The `00_setup_local_env.sh` script will attempt to install system packages via `apt-get` (requires `sudo`), clone application repos into `apps/`, install the NVIDIA HPC SDK if not present, and install Python packages (`openai`, `anthropic`).

### Step 1: Profile — Identify Hotspot Functions

```bash
# Profile any supported application (reads app_config.json for build/run settings)
bash scripts/01_profile_app.sh miniGhost
bash scripts/01_profile_app.sh XSBench
bash scripts/01_profile_app.sh LAMMPS

# miniFE uses perf-based profiling (gprof has MPI-related issues)
bash scripts/profile_miniFE_perf.sh
```

This builds the application with profiling flags (`-pg`), runs it, and generates a gprof summary JSON in `reports/<app>/profiling_summary_<timestamp>.json`.

### Step 2: LLM Analysis — Automated GPU Feasibility Assessment

```bash
# Set your API key (or create a .env file with OPENAI_API_KEY=sk-xxx)
export OPENAI_API_KEY="sk-your-key-here"
# Or: export ANTHROPIC_API_KEY="sk-ant-your-key-here"

# Run LLM analysis for any application
python3 scripts/02_analyze_hotspots.py miniGhost
python3 scripts/02_analyze_hotspots.py XSBench
python3 scripts/02_analyze_hotspots.py miniFE
python3 scripts/02_analyze_hotspots.py LAMMPS
```

This extracts hotspot function source code, constructs a detailed prompt with profiling data, calls the LLM API, and saves the analysis to `reports/<app>/llm_analysis_<timestamp>.md`. The prompt is also saved for reproducibility.

### Step 3: GPU Transformation

**miniGhost (automated OpenACC patch):**
```bash
python3 scripts/03_apply_gpu_patch.py miniGhost
```
This automatically transforms OpenMP directives to OpenACC in the Fortran source and generates a GPU makefile.

**XSBench / miniFE / LAMMPS (manual LLM-driven CUDA):**

For these applications, the GPU CUDA code was written with LLM assistance and is already included in the repository under `apps/<app>_gpu/cuda_manual/`. The CUDA kernels implement the hotspot functions identified in Steps 1–2.

### Step 4: Performance Comparison

```bash
# miniGhost — CPU (GCC+OpenMP) vs GPU (NVIDIA HPC SDK+OpenACC)
bash scripts/04_compare_miniGhost.sh

# XSBench — CPU vs CUDA across multiple lookup sizes and thread counts
bash scripts/04_compare_xsbench.sh

# miniFE — CPU CG solver vs CUDA CG solver (configurable grid size)
bash scripts/04_compare_miniFE.sh          # default 128³
bash scripts/04_compare_miniFE.sh 200      # custom grid 200³

# LAMMPS — CPU LJ force vs CUDA LJ force (configurable atom count)
bash scripts/04_compare_lammps.sh                # default 20³ FCC = 32K atoms
bash scripts/04_compare_lammps.sh 20 30 40 50    # multi-size benchmark
```

Results are saved to `reports/<app>/performance_comparison/comparison_<timestamp>.txt`.

## Two GPU Porting Strategies

This project demonstrates two distinct approaches to LLM-driven GPU acceleration:

### Strategy 1: Automated Directive-Based Porting (miniGhost)

The `03_apply_gpu_patch.py` script automatically transforms Fortran OpenMP directives to OpenACC. This approach is:
- **Fast to implement** — script-driven, no manual coding required
- **Limited in scope** — only handles directive replacement, cannot restructure data layouts or algorithms
- **Best for** — regular stencil codes with simple loop structures

### Strategy 2: Manual LLM-Guided CUDA Porting (XSBench, miniFE, LAMMPS)

For more complex applications, hotspot source code is fed to an LLM, which generates CUDA kernel implementations. The developer then integrates and benchmarks. This approach:
- **Handles complex transformations** — memory management, AoS→SoA data layout changes, kernel launch configuration
- **Requires developer expertise** — to validate, integrate, and tune the generated code
- **Best for** — irregular memory access patterns, complex data dependencies

## Configuration System

All application-specific settings (repo URLs, compilers, flags, run arguments, profiling keywords) are centralized in `app_config.json`. Scripts read this file instead of hardcoding app-specific logic, making it easy to add new applications.

Key fields per application:
- `language`, `domain` — used by LLM prompt construction
- `build.system`, `build.makefile`, `build.compiler` — build configuration
- `run.args`, `run.requires_mpi` — runtime settings
- `profiling.hotspot_keywords` — function names to extract source code for
- `gpu_patch.strategy`, `gpu_patch.gpu_compiler`, `gpu_patch.gpu_flags` — GPU build settings

## Known Issues and Notes

1. **LAMMPS source not in Git**: The LAMMPS repository is ~1 GB and is excluded via `.gitignore`. After cloning, run `bash scripts/00_setup_local_env.sh LAMMPS` to fetch it. The `LAMMPS_gpu/cuda_manual/` directory (containing the CUDA benchmark code) IS included in the repo.

2. **NVIDIA HPC SDK paths**: The `app_config.json` contains hardcoded paths to the NVIDIA HPC SDK (e.g., `/opt/nvidia/hpc_sdk/Linux_x86_64/26.1/`). If your SDK version differs, update the `gpu_compiler` paths in the config file.

3. **GPU architecture flag**: CUDA code is compiled with `-arch=sm_89` (Ada Lovelace / RTX 40-series). Change this in `app_config.json` if using a different GPU architecture (e.g., `sm_75` for Turing, `sm_86` for Ampere).

4. **miniFE profiling**: Uses `perf stat` instead of `gprof` due to MPI-related anomalies in gprof output. Run `bash scripts/profile_miniFE_perf.sh` instead of the standard `01_profile_app.sh`.

5. **WSL2 GPU access**: Ensure the Windows NVIDIA driver supports WSL2 GPU pass-through. Check with `nvidia-smi` inside WSL2.

6. **API keys**: The `.env` file is gitignored. Create it manually:
   ```bash
   echo 'OPENAI_API_KEY=sk-your-key-here' > .env
   # or
   echo 'ANTHROPIC_API_KEY=sk-ant-your-key-here' > .env
   ```

## Reports Directory

Each application's `reports/<app>/` directory contains:

- `profiling_summary_<timestamp>.json` — Parsed gprof output with hotspot rankings
- `gprof_full_<timestamp>.txt` — Full gprof output
- `llm_prompt_<timestamp>.md` — The exact prompt sent to the LLM (for reproducibility)
- `llm_analysis_<timestamp>.md` — LLM's GPU feasibility analysis
- `performance_comparison/comparison_<timestamp>.txt` — CPU vs GPU benchmark results
